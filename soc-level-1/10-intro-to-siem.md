## Core Concept

Security Information and Event Management (SIEM) is a centralized security solution used by SOC analysts to collect, process, search, correlate, and analyze logs from multiple systems.

SIEM helps analysts detect suspicious activity and investigate security incidents from one central platform.

**Basic SIEM Flow:**

`Log Sources → Log Ingestion → Parsing & Normalization → Correlation → Detection Rules → Alerts → Investigation`

---

## Log Sources

A **log source** is any device, system, or application that generates logs.

Log sources are mainly divided into two categories.

### Host-Centric Log Sources

Logs generated by activity occurring inside an endpoint or server.

Examples:

- User login attempts
- File access
- Process execution
- PowerShell commands
- Registry modifications

Common sources include Windows machines, Linux systems, and servers.

### Network-Centric Log Sources

Logs generated from communication between systems or over a network.

Examples:

- SSH connections
- FTP activity
- Web traffic
- VPN connections
- Network file sharing

Common sources include firewalls, routers, IDS/IPS, and VPN devices.

---

## Challenges of Traditional Log Analysis

Without a SIEM, analysts face several problems:

| Challenge | Problem |
|---|---|
| Numerous Log Sources | Too many devices generate large amounts of logs |
| No Centralization | Analysts must check systems individually |
| Limited Context | A single log may not reveal the complete attack |
| Limited Analysis | Humans cannot manually analyze millions of events |
| Different Formats | Different systems generate logs in different formats |

---

## Core Features of SIEM

### Centralized Log Collection

SIEM collects logs from endpoints, servers, firewalls, applications, and other systems into one central location.

Analysts no longer need to connect to every device individually.

### Parsing and Normalization

**Parsing** breaks raw logs into individual fields.

Example:

`Username = Haris | Source_IP = 10.10.5.4 | Action = Login`

**Normalization** converts logs from different sources into a consistent format.

This makes searching, correlation, and detection easier.

### Log Correlation

SIEM connects related events from different log sources.

Example:

`Unusual VPN Login → Sensitive File Access → PowerShell Execution → Outbound Connection`

Individually, these events may appear normal. Together, they may indicate compromised credentials and possible data exfiltration.

### Real-Time Alerting

SIEM uses detection rules to identify suspicious activity.

When rule conditions are matched:

`Suspicious Activity → Detection Rule Matches → Alert Triggered → SOC Analyst Investigates`

### Dashboards and Reporting

Dashboards provide a summarized view of security activity.

They may display:

- Alert highlights
- Failed login attempts
- Triggered detection rules
- Events ingested
- System health
- Top visited domains

## Common Log Sources

### Windows Logs

Windows records system activities in Windows Event Logs, which can be viewed using Event Viewer.

Each type of event has an Event ID.

Examples include:

- Login attempts
- Process execution
- Account changes
- Event log clearing

### Linux Logs

Linux stores logs in different files under `/var/log/`.

| Location | Purpose |
|---|---|
| `/var/log/httpd` | HTTP requests, responses, and errors |
| `/var/log/cron` | Scheduled cron job activity |
| `/var/log/auth.log` | Authentication events |
| `/var/log/secure` | Authentication and security logs |
| `/var/log/kern` | Kernel-related events |

**Cron Job = Linux Task Scheduler**

### Web Server Logs

Web servers generate logs containing information such as:

- Source IP address
- HTTP method
- Requested resource
- Response status code
- User-Agent

These logs can help analysts detect web attacks and suspicious requests.

---

## Log Ingestion

**Log ingestion simply means getting logs from devices and applications into the SIEM so they can be stored, searched, and analyzed.**

**Simple Example:**

`Windows Logs → Sent to SIEM → Stored and Analyzed`

### Common Log Ingestion Methods

| Method | Description |
|---|---|
| Agent / Forwarder | Lightweight software installed on endpoints that collects and sends logs |
| Syslog | Standard protocol used by devices to send logs to a central destination |
| Manual Upload | Uploading saved log files into the SIEM for analysis |
| Port Forwarding | Devices send logs to a specific port monitored by the SIEM |

### Easy Memory Trick

`Agent = Installed collector`

`Syslog = Standard log-sending method`

`Manual Upload = Upload saved logs`

`Port Forwarding = SIEM listens on a port`

## Detection Rules

Detection rules are logical conditions used by SIEM to identify suspicious activity.

**Basic Concept:**

`IF suspicious conditions are matched → THEN generate an alert`

Examples:

- Multiple failed login attempts in a short period
- Successful login after repeated failures
- Unauthorized USB device connected
- Large outbound data transfer

---

## Detection Rule Examples

### Event Logs Cleared

Attackers may clear Windows Event Logs to hide their activities.

**Detection Logic:**

`Log Source = WinEventLog AND Event ID = 104 → Alert: Event Logs Cleared`

### WHOAMI Command Execution

Attackers may execute `whoami` after gaining access to identify the current user account.

**Detection Logic:**

`Log Source = WinEventLog AND Event ID = 4688 AND Process contains whoami → Alert`

Event ID `4688` represents process creation activity.

Running `whoami` is not automatically malicious, so the analyst must investigate the surrounding context.

---

## Why Normalization is Important

Detection rules rely on specific field-value pairs.

Different systems may represent similar information differently.

Normalization converts these logs into consistent fields so SIEM rules can search and correlate events efficiently.

`Raw Logs → Parsing → Normalization → Detection Rules`

---

## Alert Investigation Workflow

When an alert is triggered, the SOC analyst investigates the related events.

`Alert Triggered → Review Detection Rule → Examine Related Logs → Gather Context → True/False Positive → Respond`

### False Positive

The activity is legitimate but triggered the detection rule.

Possible action:

- Close the alert
- Document findings
- Tune the detection rule if similar false positives occur frequently

### True Positive

The alert represents genuine malicious activity.

Possible actions:

- Investigate further
- Contact the asset owner
- Isolate the infected host
- Block malicious IP addresses
- Escalate the incident when necessary

---

## Example SIEM Investigation

Consider these events:

`Login from New VPN IP`

`↓`

`Sensitive Documents Accessed`

`↓`

`PowerShell Executed`

`↓`

`Large Outbound Connection`

A single event may not be enough to identify malicious activity.

By correlating all events, the SIEM can detect possible credential compromise and data exfiltration.

## SIEM vs EDR

| EDR | SIEM |
|---|---|
| Focuses on endpoints | Provides organization-wide visibility |
| Collects endpoint telemetry | Collects logs from many security systems |
| Detects endpoint threats | Correlates events across multiple sources |
| Can isolate hosts and terminate processes | Provides centralized searching and investigation |
| Mainly endpoint detection and response | Central security monitoring and analysis |

EDR and SIEM commonly work together.

`Endpoint → EDR → Logs/Alerts → SIEM → SOC Analyst`

| Term | Definition |
|---|---|
| SIEM | Security Information and Event Management |

| Correlation | Connecting related events from multiple sources |
| Detection Rule | Logical condition used to identify suspicious activity |
| Event ID | Identifier assigned to a specific Windows event type |
| True Positive | Correct alert representing malicious activity |
| False Positive | Legitimate activity incorrectly detected as suspicious |
| Syslog | Standard protocol used to send logs to a centralized destination |

---

## Personal Takeaways

- SIEM gives SOC analysts centralized visibility by collecting logs from many systems into one platform.
- Log ingestion simply means getting logs into the SIEM for analysis.
- Parsing breaks logs into fields, while normalization converts different log formats into a consistent structure.
- Individual events may appear harmless, but correlation can reveal a complete attack chain.
- Detection rules monitor field values and trigger alerts when suspicious conditions are met.
- A SIEM alert is only the starting point; the SOC analyst must investigate the context and determine whether it is a true or false positive.
- SIEM and EDR complement each other: EDR provides detailed endpoint visibility and response, while SIEM provides broader centralized monitoring and correlation.
